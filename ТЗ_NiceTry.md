## 0. О документе

Документ описывает **что** должно быть реализовано, а не **на чём**. Стек (язык, фреймворк, БД, хостинг) сознательно не зафиксирован — выбор остаётся за исполнителем. Единственные технические ограничения — те, что диктуются требованиями безопасности и внешними API (см. §3 и §6).

**Источники, на которых основано ТЗ:**
- Исходное ТЗ проекта (бизнес-описание, 10 разделов).
- Поставщики товаров: **два отдельных API** — **AppRoute** (gift-карты и пополнения) и **Dessly** (отправка игр / гифтов). Цены у обоих — в **USD**.
- Документация **AppRoute**: приложенный файл «AppRoute_Public_API_Documentation_RU.pdf» (AppRoute Public API v1; 12 endpoints; scopes `transaction`, `funds`, `shop`, `orders`; Base URL — плейсхолдер `{BASE_URL}`, выдаётся вместе с ключом).
- Документация **Dessly**: https://desslyhub.readme.io/reference/introduction (отдельный API для отправки игр / гифтов).
- Маппинг «категория → поставщик» задаётся в админке: по умолчанию gift-карты и пополнения — через AppRoute, отправка игр/гифтов — через Dessly.
- Готовый фронтенд-визуал проекта: файл `index.html` (эталон внешнего вида и навигации).

**Важное изменение относительно исходного ТЗ:** раздел **«События / Season Pass» полностью исключён** из объёма работ и из интерфейса. В текущем визуале (`index.html`) упоминаний этой функции нет — правки не требуются.

## 1. Краткое описание продукта

**NiceTry** — магазин цифровых товаров для аудитории РФ (15–40 лет): пополнение игровых аккаунтов, ключи и коды активации, gift-карты, подписки. Товары имеют моментальную или ручную выдачу.

**Точки входа** (в исходном ТЗ их три, работают как одно целое — один аккаунт):
1. **Сайт** — браузерная версия магазина. **Реализуется первым и является приоритетом.**
2. **Telegram-бот** — меню, уведомления, запуск WebApp. **Более поздний этап.**
3. **Telegram WebApp** — тот же магазин внутри Telegram. **Более поздний этап.**

На текущем этапе разрабатывается **только сайт**. Бот и WebApp вынесены в отдельный поздний блок (§5.7), но архитектуру аккаунта и заказов нужно проектировать так, чтобы их подключение не требовало переделки ядра.

## 2. Глоссарий

| Термин | Значение |
|---|---|
| **Поставщик AppRoute** | Внешний сервис, у которого магазин **закупает** gift-карты и пополнения через API: voucher-коды (shop) и прямые пополнения (DTU). Цены в USD. Документация — приложенный PDF. |
| **Поставщик Dessly** | Отдельный поставщик/API для **отправки игр (гифтов)**. Цены в USD. Документация: desslyhub.readme.io. Доступ по своему API-ключу (предоставит заказчик). |
| **Платёжная система (эквайринг)** | Внешний сервис, через который магазин **принимает деньги от покупателей** (рубли, криптовалюта). Это **другой** сервис, не AppRoute. |
| **Баланс магазина у поставщика** | Средства магазина в AppRoute (по умолчанию в USD), с которых списывается стоимость исполнения заказов. |
| **Баланс пользователя** | Внутренний кошелёк покупателя в магазине (в ₽), которым можно оплачивать заказы. |
| **Тип выдачи** | Одна из 4 моделей товара: моментальная, пополнение-авто, пополнение-ручное, ручная выдача. |
| **shop / DTU** | Типы заказа в AppRoute: `shop` — покупка кода/ваучера; `dtu` — прямое пополнение без кода. |
| **denominationId** | Идентификатор конкретного номинала товара в каталоге поставщика; передаётся при создании заказа. |
| **referenceId** | Ключ идемпотентности заказа в AppRoute (уникальный, 1..40 символов). |
| **X-API-Key** | Секретный ключ доступа к AppRoute. Хранится **только на сервере**. |

## 3. Архитектура и денежные потоки (ключевой раздел)

В проекте существуют **два независимых денежных контура**. Их важно не смешивать — от этого зависит порядок этапов и пауза.

### Контур A — Покупатель → Магазин (приём оплаты)
Покупатель платит магазину рублями / криптой / с внутреннего баланса. Для приёма реальных рублей и крипты нужен **договор с платёжной системой (эквайринг)** и её верификация. **До верификации реальный приём денег невозможен** — именно поэтому в плане предусмотрена пауза (см. §7).

### Контур B — Магазин → Поставщик (исполнение заказа)
Чтобы выдать покупателю код или провести пополнение, магазин обращается к **AppRoute API** и тратит **свой** баланс у поставщика. Этот контур **не зависит** от платёжной системы и может быть полностью реализован и протестирован заранее (каталог, заказы, выдача), включая пополнение баланса магазина у поставщика через `funds/invoices`.

### Принципы
- **Разделение слоёв:** публичная витрина (сайт) ↔ серверный бэкенд (бизнес-логика, секреты, интеграции) ↔ админ-панель.
- **Все секреты (X-API-Key, ключи эквайринга) — только на сервере.** Фронтенд никогда не обращается к AppRoute напрямую.
- **Идемпотентность и асинхронность** заложены в работу с заказами (см. §5.4).

```
[ Покупатель ]
     │  оплата ₽ / крипта (Контур A — через платёжную систему)
     ▼
[ Сайт / WebApp ] ── HTTP ──▶ [ Бэкенд магазина ] ──X-API-Key──▶ [ AppRoute API ]
     ▲                              │  (Контур B — закупка/выдача)        │
     └──────── выдача кода/статус ──┘◀────────── voucher / attributes ────┘
```

## 5. Функциональные требования

### 5.1 Витрина (сайт)
Эталон внешнего вида и навигации — `index.html`.
- **Главная:** сменяемый баннер с переходом в Telegram-группу (управляется из админки), глобальный поиск, лента категорий, сетка каталога, секции «Популярное», «Новинки», «Пополнения / По играм».
- **Каталог / категория:** фильтры (категория, тип выдачи, наличие, цена), сортировка, поиск, карточки, пагинация / «показать ещё», хлебные крошки.
- **Карточка товара — 4 варианта по типу выдачи** (см. §5.3).
- **Корзина:** список позиций, изменение количества, промокод, итог, переход к оплате. Товары с моментальной выдачей можно купить сразу с карточки.
- **Оформление заказа:** список товаров, промокод (%/фикс), поле «данные аккаунта» (логин/ID, чекбокс «сохранить в профиле»), выбор способа оплаты (баланс / рубли / крипта), итог со скидкой, кнопка «Оплатить».
- **Экран успеха:** детали заказа; для моментальной выдачи — показ ключа/кода с копированием.
- **Авторизация:** по email (magic link) и через **Telegram**. Экран «введите email» + экран «письмо отправлено»; вход через Telegram использует привязанный аккаунт. Пароль не используется.

### 5.2 Профиль
- **Баланс** и пополнение (выбор способа: рубли / крипта, ввод суммы).
- **Мои заказы** — список с фильтром по статусу (новый / оплачен / выдан / отменён), просмотр выданных ключей/кодов.
- **Привязка Telegram** (код привязки в обе стороны; конфликт привязки запрещён и сопровождается предупреждением).
- **Статус пользователя** — текущий статус и его скидка (назначается админом вручную).
- **Реферальная программа** — личная ссылка, статистика, начисленные бонусы.
- **Поддержка** (ссылка в Telegram), **Документы** (оферта, политика конфиденциальности).

### 5.3 Типы товаров (4 модели)

| Тип | Поведение для покупателя | Откуда берётся / как создаётся в админке |
|---|---|---|
| **1. Моментальная выдача** | Код/ключ показывается сразу после оплаты, дублируется в уведомлении. | **(а) Из файла:** админ загружает текстовый файл, каждая строка = единица товара; при покупке выдаётся одна строка; когда строки кончились — товар авто-скрывается; видны остатки, можно дозагрузить. **(б) От поставщика:** позиции тянутся из AppRoute (`shop`); админ включает/выключает позицию и ставит свою цену поверх цены поставщика; если поставщик отметил позицию недоступной — авто-скрытие. |
| **2. Пополнение (авто)** | Выбор региона (если есть), ввод суммы в ₽, показ актуального курса и расчёт зачисления. | AppRoute `dtu`. Админ задаёт название, регионы, поставщика и источник курса, мин/макс сумму. Конвертация и отправка запроса — автоматически. |
| **3. Пополнение (ручное)** | Ввод суммы в ₽ без курса; пометка «обработает менеджер». | Админ задаёт только название и описание; после оплаты заказ уходит менеджеру как ручная выдача. |
| **4. Ручная выдача** | Фиксированная цена, указан срок выдачи; «заказ уйдёт менеджеру». | Стандартный товар с фикс-ценой; после оплаты — менеджеру. |

Карточка любого товара: название, описание, тип выдачи, наличие, отзывы/похожие. Для пополнений вместо фикс-цены — поле ввода суммы (+ курс для авто).

**Категории на старте** (источник — AppRoute, кроме отправки игр через Dessly): PSN gift card, Steam gift card, Rockstar, Roblox, App Store, Google, WoW Time Card, Blizzard, Minecraft, отправка игр / гифтов (Dessly).

**Ценообразование.** Наценка задаётся **под каждую категорию отдельно** и управляется в админке; курс USD→₽ также задаётся в админке. Формула:

`цена_₽ = цена_поставщика_USD × курс × (1 + наценка%)`

**Пример (Steam gift):** наценка 14%, курс 80. Игра за 10$ → 10 × 80 = 800 ₽ → +14% = **912 ₽**.

Конкретные значения наценки и курса по каждой категории задаёт администратор; стартовые значения предоставляет заказчик по мере наполнения каталога.

**Реферальная программа:** реферальный процент **~12–15% для автоматических товаров** (начисляется рефереру с покупок приглашённого; настраивается в админке). Для остальных типов — уточняется (см. §8).

### 5.4 Интеграция с поставщиками (AppRoute + Dessly)

**Регистрация и токен (AppRoute).** Для доступа к API AppRoute необходимо **зарегистрироваться на стороне поставщика и получить ключ `X-API-Key`** с нужными scopes (`transaction`, `funds`, `shop`, `orders`). Да — это именно регистрация на сайте поставщика ради создания токена. Токен и Base URL (`{BASE_URL}`) предоставляет владелец проекта **по запросу исполнителя** на этапе интеграции; до этого работа ведётся на заглушках/моках. Ключ хранится только на сервере.

**Второй поставщик — Dessly** (отправка игр / гифтов) — **отдельный API** со своей документацией: https://desslyhub.readme.io/reference/introduction. Доступ по своему API-ключу (предоставит заказчик). Маппинг «категория → поставщик» хранится в админке: по умолчанию gift-карты и пополнения — через AppRoute, отправка игр/гифтов — через Dessly.

Оба API независимы: AppRoute описан в приложенном PDF (`AppRoute_Public_API_Documentation_RU.pdf`), Dessly — на desslyhub.readme.io. Клиент к каждому реализуется по своей документации.

**Базовые правила (из документации):**
- Все data-plane запросы требуют заголовок `X-API-Key`; для POST с телом — `Content-Type: application/json` (иначе 415).
- Суммы у поставщика по умо����чанию в **USD**; даты — ISO-8601; поля в camelCase.
- `statusCode` — внутренний код приложения, **не равен** HTTP-статусу; проверять и HTTP, и `status`, и `statusCode`; всегда логировать `traceId`.
- **Идемпотентность:** в `POST /orders` поле `referenceId` обязательно и уникально; повтор того же `referenceId` вернёт первый результат (`IDEMPOTENCY_REPLAY`). Генерировать новый UUID на каждый реальный заказ.
- **Асинхронность:** `POST /orders` может вернуть `202 / IN_PROGRESS`. Терминальные статусы: `SUCCESS`, `PARTIALLY_COMPLETED`, `CANCELLED`. Дожидаться результата через `GET /orders` по `orderId`/`referenceId`, polling с 1–2 сек и exponential backoff.
- **Маскирование кодов:** `GET /orders` по умолчанию маскирует voucher-коды; полный код — только с `orderId`/`referenceId` и `unhide=true`. `unhide=true` имеет побочный эффект (код помечается как полученный) — раскрывать только в момент фактической выдачи покупателю.

**Используемые endpoints:**

| Endpoint | Назначение в проекте |
|---|---|
| `GET /api/v1/accounts` | Баланс магазина у поставщика (контроль средств перед закупкой). |
| `GET /api/v1/accounts/transactions` | Сверка движений средств. |
| `GET /api/v1/funds/methods` | Способы пополнения баланса магазина у поставщика. |
| `GET` / `POST /api/v1/funds/invoices` | Создание/мониторинг инвойсов на пополнение баланса магазина. |
| `GET /api/v1/services`, `GET /api/v1/services/{id}` | Каталог поставщика: сервисы, номиналы (`denominationId`), цены, наличие, обязательные `fields`. Источник импорта товаров типов 1(б) и 2. |
| `GET /api/v1/steam-currency/rates` | Курсы Steam-валют относительно USD — для калькулятора пополнений (тип 2). |
| `POST /api/v1/orders` (shop) | Покупка voucher-кода → результат в `result.vouchers`. |
| `POST /api/v1/orders` (dtu) | Прямое пополнение → результат в `result.attributes`. |
| `POST /api/v1/orders` (dtu, `checkOnly=true`) | Pre-check DTU до оплаты: валидность аккаунта, цена, nickname. |
| `GET /api/v1/orders` | Polling статусов, история, безопасное раскрытие кодов (`unhide`). |

**Сценарии (см. документацию, разделы 5/B/C):**
- **Покупка кода (shop):** выбрать `denominationId` → `POST /orders` (shop, уникальный `referenceId`) → при 202 polling `GET /orders` → дождаться терминального статуса → `unhide=true` для выдачи кода покупателю.
- **Пополнение (DTU):** прочитать `fields` сервиса → `checkOnly=true` (quote + проверка аккаунта) → `POST /orders` (dtu) → polling → результат из `result.attributes`.
- **Пополнение баланса магазина:** `funds/methods` → `funds/invoices` (создать) → оплатить → мониторинг → `accounts`.

**Обработка ошибок (минимум):** `OUT_OF_STOCK (9)`, `INSUFFICIENT_FUNDS (10)`, `VALIDATION_ERROR (3)`, `UNAUTHORIZED/FORBIDDEN (4/5)`, `LIMIT_REACHED (8)`, `UPSTREAM_ERROR (11)`, `INTERNAL_ERROR (12)` — каждому соответствует понятное состояние в UI и/или ретрай-политика.

### 5.5 Приём платежей от покупателя (Контур A)
- **Платёжная система (рубли): Pay4game.** Онлайн-касса есть; **фискальные чеки (54-ФЗ) не требуются**.
- Способы оплаты на оформлении: **внутренний баланс**, **рубли** (на запуске); **криптовалюта** — позже (Этап 7).
- Пополнение баланса пользователя: на запуске рублями; крипта добавляется позже.
- Боевое подключение Pay4game выполняется **после верификации** (см. §7). До этого — UI и серверные «хуки» оплаты на моках/песочнице, без реального списания.

### 5.6 Админ-панель
Закрытый веб-интерфейс с боковым меню. Разделы (исходные 8.1–8.10, **без Season Pass**):
1. **Каталог** — список товаров, поиск/фильтры, скрытие без удаления, создание товара одного из 4 типов (§5.3), управление категориями, остатки/дозагрузка файла, цены поверх поставщика, авто-скрытие.
2. **Заказы** — таблица с фильтрами (дата, статус, тип, пользователь); статусы новый/оплачен/выдан/отменён; ручной перевод в «выдан» (→ уведомление покупателю); отмена с возвратом средств на баланс.
3. **Пользователи** — поиск по имени/email/Telegram; контакты, баланс, история, реф-статистика, назначение/снятие статуса; ручная корректировка баланса с указанием причины.
4. **Промокоды** — список и создание (произвольный код или автоген, фикс/процент, лимит использований, срок), статистика использования.
5. **Баннеры** — загрузка изображения и ссылки, порядок, вкл/выкл.
6. **Реферальная программа** — проценты отдельно для моментальной и ручной выдачи; применяется к новым заказам.
7. **Статусы пользователей** — список статусов и их скидок; добавление/изменение/удаление.
8. **Аналитика** — дашборд с выбором периода (сегодня / 7 / 30 дней / произвольный): выручка, заказы, средний чек, новые пользователи, пополнения; график выручки по дням (переключение выручка/кол-во); продажи по категориям; топ-10 товаров; способы оплаты; реф-статистика; промокоды; блок UTM.
9. **UTM-метки** — генерация пары ссылок (бот + сайт), учёт переходов/регистраций/выручки, фильтры, деактивация.
10. **Рассылки** — текст + изображение + кнопка, сегментация (все / с заказами / без заказов / по статусу / по UTM), отправка сразу или по расписанию, предпросмотр размера аудитории, статистика, статусы (черновик/запланирована/отправляется/завершена).

> **Зависимость:** рассылки и уведомления физически работают через Telegram-бота, поэтому их «боевая» часть включается вместе с блоком бота (§5.7). В админке интерфейс может быть готов раньше.

### 5.7 Telegram-бот и WebApp (поздний блок)
- **Бот:** приветствие + документы (оферта/политика) с подтверждением ознакомления; меню (Мои заказы, Код привязки, Отзывы, Поддержка, Открыть магазин); авто-авторизация в WebApp.
- **WebApp:** тот же магазин (переиспользует фронтенд и API сайта), авторизация через Telegram.
- **Единый аккаунт** сайта и Telegram через привязку.

### 5.8 Уведомления
Через бота: **товар выдан** (моментально — сразу, ручной — при закрытии менеджером) и **запрос отзыва** спустя время после выдачи — обязательны. Уведомления «заказ принят» / «оплачен» — опциональны (по решению заказчика их можно не отправлять). Включается вместе с блоком бота.

### 5.9 Отзывы
После закрытого заказа бот просит оставить отзыв; отзывы публикуются в отдельном Telegram-канале (кнопка «Отзывы»). Включается вместе с блоком бота.

### 5.10 Реферальная про��рамма, статусы, промокоды
Логика начислений (процент по типам товаров), скидки статусов, применение промокодов на оформлении. Значения процентов и скидок — конфигурируемые (см. §8, открытые вопросы).

### 5.11 UTM-метки и рассылки
См. §5.6 (9) и (10). Учёт источника при регистрации и по существующим пользователям.

### 5.12 Системные состояния
404, пустые состояния, загрузка (скелетоны), ошибки форм, тосты-уведомления, «нет в наличии», состояния кнопок (hover/active/disabled/loading). Эталон — `index.html`.

## 6. Нефункциональные требования
- **Безопасность:** все внешние ключи (`X-API-Key`, ключи эквайринга) — только на сервере; фронт не имеет к ним доступа. Разграничение доступа к админке. Защита данных аккаунтов покупателей.
- **Идемпотентность и надёжность:** уникальный `referenceId` на заказ; корректная обработка `IDEMPOTENCY_REPLAY`; устойчивый polling с backoff; учёт частичного выполнения (`PARTIALLY_COMPLETED`).
- **Согласованность UI:** единая дизайн-система из `index.html` (цвета, отступы, компоненты, типографика).
- **Адаптивность:** десктоп + мобайл (бургер-меню, мобильные корзина/каталог/фильтры).
- **Доступность:** контраст, фокус, читаемые размеры, кликабельные зоны.
- **Локализация:** интерфейс на русском; цены покупателю — в ₽ (разделитель тысяч — пробел: «1 499 ₽»).
- **Логирование/наблюдаемость:** хранить `traceId` ответов AppRoute вместе с заказом для поддержки.
- **Правовое:** оферта и политика конфиденциальности доступны на сайте и в боте.
- **Репозиторий:** весь код ведётся в Git-репозитории **`ivantemshik/NiceTry`** (GitHub); изменения коммитятся туда сразу. Рабочая папка проекта: `C:\\Users\\user\\Desktop\\NiceTry`.
- **Деплой:** автоматический деплой в **Vercel** из GitHub (push в основную ветку → prod, PR → preview). На раннем этапе допускается локальный запуск через **`.bat`-скрипт** для проверок и тестов. Секреты (`X-API-Key`, ключ Dessly, ключи Pay4game) хранятся в переменных окружения Vercel / локального окружения, не в репозитории.
- **Фронтенд-эталон:** файл `index.html` помещается в репозиторий (корень или `public/`) как стартовый фронтенд и эталон вёрстки.

- **Скидки статусов:** на старте — **захардкодить плейсхолдерные значения** (редактируются в админке).
- **Мин/макс суммы** заказа и пополнения: на старте — **захардкодить плейсхолдерные значения** (редактируются в админке).
- **Возвраты:** на **внутренний баланс** пользователя.
