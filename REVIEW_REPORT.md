**Про `APPROUTE_BASE_URL` (важно).** В задании просили «найти Base URL в PDF и подставить». **В PDF реального
URL нет** — документация AppRoute использует плейсхолдер `{{BASE_URL}}` (стр. 11 и 119: «API доступен по
`{{BASE_URL}}`»), реальный адрес data-plane выдаётся поставщиком **вместе с ключом**. Поэтому в `.env.local`
оставлен понятный плейсхолдер `APPROUTE_BASE_URL=your_approute_base_url` + TODO. Пока он плейсхолдер, клиент
работает на моках; как только вставите реальный URL и ключ — боевой режим включится автоматически, без
изменения кода. Подставлять выдуманный URL было бы вредно (сломало бы переключение в бой).

### ⚠️ Требуется действие владельца (1 шаг)
На живой БД **схема применена** (seed прошёл), но проверка показала: анонимный ключ **не читает** `products`/
`categories` (0 строк), а service-role читает (видит 34 товара). Значит RLS включён, **но публичные
read-политики ещё не применены** — файл `supabase_security.sql` не выполнен в БД.

**Что сделать:** открыть Supabase → SQL Editor → выполнить **`supabase_security.sql`** (идемпотентно).
Он добавит:
- публичное чтение активных `products`/`categories`/`banners`/`user_statuses`;
- защиту закрытых таблиц (voucher-коды, промокоды и т.д.) — только service-role;
- политику `users_insert_self` и триггер защиты полей (`is_admin`/`balance`/`status_id`) от подмены.

До этого витрина работает через фолбэк-каталог (товары видны, но из JSON, не из БД). После — публичная
витрина начнёт читать товары из БД. *(Применить из кода нельзя: service-ключ через PostgREST не выполняет DDL —
это разовая операция в дашборде.)*

## 5. Что НЕ делалось (по требованию ПАУЗЫ)
Приём реальных платежей (Контур A: Pay4game, карты, крипта) — **не реализуется** до верификации и передачи
ключей эквайринга заказчиком (ТЗ §7). В коде это уже отражено: методы оплаты `card`/`crypto` возвращают
HTTP 501; работает только оплата с внутреннего баланса.
