### Шаг 4. Найденные и исправленные баги
1. **Цены off-by-one (FP)**: `800*1.14 = 912.0000000000001 → ceil = 913`. Формула в `catalog.ts` и
   `seed.mjs` переписана на целочисленный множитель `(100+markup)/100`. Теперь 912/456 как в ТЗ. БД пересеяна.
2. **Admin API на сессионном клиенте под RLS** (критично): все `/api/admin/*` читали/писали через
   anon-клиент → под RLS списки users/orders/promo были бы пусты, создание/правка/возврат — запрещены
   или молча откатаны триггером. Введён гард `requireAdmin()` (сессия→проверка is_admin→service-role)
   во всех 11 admin-роутах + дашборд переведён на service-role (доступ к странице уже закрыт layout).
3. **Профиль нового пользователя не создавался**: INSERT шёл через сессионный клиент и зависел от
   политики `users_insert_self` (на живой БД её нет) → 500. Перенесён на service-role после auth.
4. **PATCH /admin/users/[id]: дельта баланса считалась после апдейта** → всегда 0, транзакция
   корректировки не писалась. Исправлено: читаем баланс до апдейта, знаковая сумма, требуем причину.

## Этап 6 — Telegram-бот + Mini App (WebApp с сайтом) — 2026-06-01

Реализованы Telegram-бот и Mini App (Telegram WebApp), открывающий сайт, по ТЗ §5.7/§5.8. Бот подключён к реальному Telegram API (@asdadawdawdadbot, id 8727167694), настройка выполнена программно.

### Архитектурное решение (без DDL)
Пароль Postgres недоступен → DDL не выполнялся. Спроектировано **без новых таблиц**:
- Каноническая связь — существующее поле `users.telegram_id` (UNIQUE) + `telegram_username`.
- Привязка аккаунтов — **stateless HMAC-токены** (подписаны токеном бота, TTL 15 мин), бинарно упакованы, чтобы влезать в deep-link ≤64 символа. БД для кодов не нужна, подделать/переиспользовать нельзя.
- Идемпотентность уведомлений — через **атомарные переходы статусов заказа** (как в refund): уведомление шлётся ровно в момент флипа, повторные webhook/PATCH дублей не дают.
- Запрос отзыва «спустя время» — маркер в существующей таблице `reviews` (rating=NULL, comment='review_requested') + cron-эндпоинт.

### Бот (вся логика)
- `src/lib/telegram/config.ts` — env/константы.
- `src/lib/telegram/verify.ts` — проверка HMAC-подписи `initData` (алгоритм Telegram WebApp) + свежесть auth_date; подписанные токены привязки (site→tg deep-link, tg→site claim).
- `src/lib/telegram/client.ts` — клиент Bot API: ретраи (429 retry_after, сетевые сбои), таймауты, `TelegramApiError.isBlocked` (403).
- `src/lib/telegram/account.ts` — единый аккаунт: `ensureTelegramUser` (telegram-first, без дублей при гонке), `linkTelegramToUser` (email-first; идемпотентно; конфликт → предупреждение; пустой авто-аккаунт → безопасное слияние).
- `src/lib/telegram/session.ts` — выдача сессии Supabase по email без письма (generateLink+verifyOtp).
- `src/lib/telegram/bot.ts` — `processUpdate`: /start [token], /help, /menu, /link; меню (Открыть магазин = Mini App, Мои заказы, Код привязки, Отзывы, Поддержка); callback link_code/docs_ack; устойчив к повторам/ошибкам.
- `src/lib/telegram/notify.ts` — уведомления: выдан / запрос отзыва / пополнение / возврат. Best-effort.

### API-роуты
- `POST /api/telegram/webhook` — приём апдейтов (проверка X-Telegram-Bot-Api-Secret-Token; всегда 200).
- `POST /api/telegram/auth` — Mini App авто-авторизация: проверка initData → единый аккаунт → сессия (cookies).
- `POST /api/telegram/link-token` — deep-link привязки (email-first).
