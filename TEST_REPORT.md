### 3.2 🔴 КРИТИЧЕСКАЯ уязвимость — найдена, исправление готово, требует 1 шага владельца

**V-1 — Повышение привилегий и накрутка баланса прямым запросом к БД.**

- **Суть:** базовая схема создаёт RLS-политику `users_update_own` (`FOR UPDATE USING auth.uid()=id`) **без
  ограничения по колонкам**. Anon-ключ публичен (уходит в браузер). Любой залогиненный пользователь может
  прямым REST-запросом (`anon-ключ + свой JWT`, минуя приложение) выполнить
  `UPDATE users SET is_admin=true, balance=999999 WHERE id=<свой>` — и стать админом / начислить себе деньги.
- **Доказано:** тест-зонд выполнил эту атаку против боевой БД — `is_admin` стал `true`, `balance` стал `999999`.
- **Причина:** файл `supabase_security.sql` (с триггером защиты полей) **не применён** к боевой БД
  (тот же симптом — аноним не читает `products`/`categories`, т.е. публичные read-политики тоже не применены).
- **Исправление (в репозитории):** `supabase_security.sql` усилен — теперь **удаляет** `users_update_own`
  (приложение никогда не меняет `users` со стороны клиента — все мутации через service-role) **и** оставляет
  триггер `protect_sensitive_user_fields` как второй рубеж.
- **Как закрыть (1 команда, нужен пароль БД из дашборда Supabase → Settings → Database):**
  ```bash
  $env:SUPABASE_DB_URL="postgresql://postgres:PASSWORD@db.<ref>.supabase.co:5432/postgres"
  npm run db:secure
  ```
  Либо: Supabase → SQL Editor → выполнить `supabase_security.sql` (идемпотентно).
- **Почему не закрыто автоматически:** применение требует **DDL**, а у тест-раннера есть только API-ключи
  (service-role через PostgREST DDL не выполняет) — **пароля БД нет**. Это операционный шаг владельца.
- **Статус теста:** проверка «прямое повышение привилегий заблокировано» помечена `skip` с явным обоснованием
  и громким предупреждением `[CRITICAL]` в выводе. После `npm run db:secure` тест автоматически становится
  активным и зелёным.

> Замечание: **уровень приложения уже безопасен** — PATCH профиля принимает только `telegram_id` (доказано
> тестом), фронт не делает клиентских апдейтов `users`. Дыра — исключительно на уровне конфигурации БД.

## 5. Что ФИЗИЧЕСКИ нельзя проверить без реальных ключей / оплаты (за ПАУЗОЙ)

1. **Боевой AppRoute API** — нет реального `APPROUTE_BASE_URL` (в PDF только плейсхолдер `{{BASE_URL}}`; ключ есть,
   адрес data-plane выдаётся с ключом). Протестирован мок, форма ответов идентична боевой. Не проверены на боевом:
   реальная выдача voucher/DTU, `funds/invoices` (реальное пополнение баланса магазина), живые `steam-currency/rates`,
   фактические коды/`traceId` поставщика, реальные тайминги polling.
2. **Боевой Dessly API** — `DESSLY_API_KEY` плейсхолдер; реальный base URL и схема авторизации (Bearer vs X-API-Key)
   не подтверждены. Протестирован мок.
3. **Pay4game (Контур A, приём денег от покупателя)** — за ПАУЗОЙ по ТЗ §7. card/crypto возвращают 501.
   Реальный приём рублей/крипты, вебхуки оплаты, верификация — не реализуются и не тестируются до передачи ключей.
4. **Реальная доставка email magic-link** — зависит от боевого SMTP. Локально используется dev-login (без письма).
5. **Telegram бот / WebApp / уведомления / боевые рассылки / отзывы** — Этап 6, вне объёма до ПАУЗЫ.
6. **Закрытие V-1 (DDL)** — требует пароля БД (есть только у владельца). Команда готова: `npm run db:secure`.

## 6. Функциональное замечание (не блокер, не уязвимость)

Публичные read-политики (`products`/`categories`/`user_statuses`) **не применены** на боевой БД (тот же
`supabase_security.sql`). Поэтому витрина читает каталог из **JSON-фолбэка** (`buildCatalogProducts`) — товары и
**цены корректны** (смоук: Steam $5 = 456 ₽, 10 категорий), но не из таблиц БД. После `npm run db:secure` витрина
начнёт читать из БД (политики публичного чтения добавятся тем же файлом). Логику переписывать не нужно.
